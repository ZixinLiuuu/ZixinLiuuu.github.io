Exploratory Data Analysis

Dataset 1

We downloaded our dataset 1 from the UC Irvine Machine Learning Repository. We will use this dataset to generate a social network which is a page-page graph of verified Facebook sites. Hence, there will be lots of nodes in this social network, and these nodes represent official Facebook pages while the links are mutual “likes” between sites. Node features are extracted from the descriptions of the site that the page owners created to summarize the purpose of the site. This graph was collected through the Facebook Graph API in November 2017 and restricted to pages from 4 categories which are defined by Facebook. These categories are: politicians, governmental organizations, television shows and companies. Additionally, this dataset comprises five attributes: 'id,' 'Facebook id,' 'page name,' 'page type,' and 'edge.' Four of these attributes—'id,' 'Facebook id,' 'page type,' and 'edge'—have no data issues. The primary data concern arises from the 'page name' attribute. Although there are no missing values in this column, it contains values that encompass various language types, including English, Chinese, Japanese, and Thai. The following is a more detailed explanation of our data quality issues and preprocessing steps.

Handling "Bad Bytes": The initial dataset exhibits unreadable characters when using the "latin-1" encoding method. Further examination reveals that these unreadable characters are symbols from various languages, which are part of the text. We replace these unreadable characters with spaces, as they don't compromise text readability or usability.

Missing Values: Several rows contain missing values in the form of NaN entries or solely continuous spaces in specific columns. After addressing the "Bad Bytes" issue, we filter the DataFrame to remove rows with NaN values in the 'page_name' column. For detecting rows that consist solely of continuous spaces in the 'page_name' column, we employ the Python "re" library to match the rows that we need.

Organizing Data Context: The dataset displays inconsistencies in capitalization. Furthermore, the original data is disorganized. We posit that restructuring and categorizing the data into meaningful groups will not only aid in identifying patterns within similar data but also significantly improve the computational efficiency of subsequent data analysis. After handling missing values, we convert the text in the "page_name" column to uppercase for consistency. Finally, we sort the data based on two columns: "page_name" and "page_type" to enhance data organization.

Dataset 3

Data Collection and Preprocessing

Data Collecting

News involving the keyword "Amazon" on google.com during a specific time period (2016-1-1 to 2019-12-31) [API]. We used a news API called GNews, deployed and ran it in the code space of GitHub, adjusted the parameters, set the keyword to "Amazon", and the time span was from ‘2016-1-1’ to ‘2019-12-31’. Finally, store the obtained data in a CSV file. It contains five attributes: title, news description, publication date, URL, and publisher. The time node of the API is in days, but the output data will contain a very detailed release time, accurate to the hour, minute, and second.

Data Quality Issues and Preprocessing

  • Duplicate Header Row: Due to the format of the news API when extracting data, the original data has multiple title rows, that is, there are multiple repeated "title, description, published date, url, publisher" rows. – Filtered out rows with the title as "title" but retained the header and this step deleted 274 rows.
  • Duplicate Data Row: Due to the format of the news API when extracting data, there are many duplicate rows. – This step deleted 2166 rows. Duplicate rows are deleted, and only one piece of non-duplicate data is retained.
  • Same Name: Because the Amazon company and the Amazon Rainforest have the same name, and since only the keyword "Amazon" was searched when extracting the data, it is not sure whether the collected data contains news data about the Amazon Rainforest, so some filtering needs to be done here. -- Distinguished between the Amazon company and the Amazon rainforest, and this step deleted 114 rows.
  • Abnormal Row: Since only data from 2016 to 2019 was selected when obtaining the data set, the data may only come from 2016 to 2019. However, when checking the data set, it was found that years other than 2016-2019 appeared. data, so this data falls outside of our desired timeframe and will be excluded. -- Removed rows where the year was not between 2016-2019, and this step deleted 62 rows.
  • Decomposed Date: This step added four attributes. This decomposition is useful for later plotting histograms over time.
  • Text Cleaning: This step converts text into tokens. This makes it easier to draw the histogram later. Filter punctuation, single letters, and stop words. Clean the title column and store it in the title_tokens column. The description column is not cleaned here because most of the description data is the same as the title data, so only the title column is taken here for processing.
